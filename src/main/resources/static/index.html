Spring Boot 静态资源目录规则

"By default, Spring Boot serves static content from a directory called /static (or /public or /resources or /META-INF/resources) in the classpath."

— Spring Boot Reference Documentation · Static Content

1. 为什么是 src/main/resources/ 下？

关键词是 "in the classpath"。Maven 标准布局规定： 

src/main/resources/  →  编译/打包后放入 JAR 的 classpath 根路径
src/main/java/       →  编译后的 .class 文件放入 classpath 根路径

因此映射关系为：

src/main/resources/static/   →  classpath:/static/   →  HTTP 可访问 ✓
src/main/resources/public/   →  classpath:/public/   →  HTTP 可访问 ✓
src/main/resources/data.sql  →  classpath:/data.sql  →  HTTP 不可访问 ✗

src/main/ 本身不是 classpath；src/main/webapp/ 仅在 WAR 打包时有效，JAR 部署请勿使用（官方文档 TIP 明确说明）。

2. 四个目录的行业惯例

目录 访问路径 行业用途 推荐程度
static/ /** 主力目录，绝大多数项目只用这一个 ⭐ 首选
public/ /** 完全公开资源：robots.txt、favicon.ico、登录页（Spring Security 白名单语义更清晰） 可选
resources/ /** 历史遗留，与 Maven resources 概念同名，容易混淆，几乎不用 不推荐
META-INF/resources/ /** JAR 包作者专用（Knife4j / Webjars 把 UI 资源打进依赖 JAR 时放这里） 业务代码勿用

3. 主流做法对比

方案 A：只用 static/（最常见）

static/
├── css/
├── js/
├── images/
├── index.html
├── favicon.ico
└── robots.txt

99% 的 Spring Boot 项目只用 static/，其余三个目录留空或不创建。

方案 B：static/ + public/ 按语义区分（适合有权限控制的项目）

static/             ← 需登录后才能访问的页面
├── dashboard.html
├── app.js
└── style.css

public/             ← 完全公开、无需认证的资源
├── favicon.ico
├── robots.txt
├── login.html
└── register.html

配合 Spring Security 白名单配置语义更清晰：

http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/public/**").permitAll()
    .anyRequest().authenticated()
);

方案 C：META-INF/resources/（库作者专用）

普通业务开发者不应手动往这里放文件。这是给打包成 JAR 的前端库使用的，例如 Knife4j、Bootstrap Webjars。

4. 本项目建议

本项目所有静态文件均位于 static/，这是正确且主流的做法。 若后续引入 Spring Security，可将登录/注册/忘记密码页移至 public/， 使权限白名单配置的语义更直观。
